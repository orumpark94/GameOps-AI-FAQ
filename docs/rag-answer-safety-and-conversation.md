# RAG 답변 안전성과 연속 질문 설계

## 1. 목적

GameOps AI FAQ 챗봇은 검색된 FAQ에 근거가 있을 때만 답변하고, 적절한 문서를 찾지 못하면
고객센터 문의를 안내한다.

```text
사용자 질문
→ Knowledge Base Retrieve
→ 관련도 점수 검사
→ 통과: Nova Micro 답변 생성
→ 탈락: 고정 고객센터 안내
```

## 2. 관련도 점수 기반 답변 제한

Amazon Bedrock Knowledge Bases의 `Retrieve` API는 검색된 문서 chunk와 관련도 점수를 반환한다.
chatbot-api는 가장 높은 점수가 설정된 임계값 이상인지 검사한다.

```text
BEDROCK_RETRIEVAL_SCORE_THRESHOLD=0.6
```

점수가 없거나 임계값보다 낮으면 LLM을 호출하지 않고 다음 고정 응답을 반환한다.

```text
사용자님의 질문에 적절한 답변이 없습니다.
자세한 사항이 궁금하시다면 sjpark@hanbitsoft.com으로 메일 부탁드립니다.
```

`0.6`은 초기값이며 정확도를 의미하지 않는다. 실제 FAQ 질문, FAQ에 없는 질문, 엉뚱한 질문의
점수를 수집한 뒤 조정해야 한다.

## 3. 답변 생성

점수를 통과한 경우 chatbot-api는 검색된 문서 본문만 Nova Micro의 `Converse` API에 전달한다.

```text
검색에 사용한 문서
→ chatbot-api가 context로 구성
→ Nova Micro
→ 답변과 검색 문서 출처 반환
```

검색과 답변 생성을 분리한 이유는 다음과 같다.

- 어떤 문서를 모델에 전달했는지 chatbot-api가 통제할 수 있다.
- 검색을 중복 수행하지 않는다.
- 점수가 낮은 질문은 모델 호출 전에 차단할 수 있다.
- 검색 정책과 생성 프롬프트를 분리할 수 있다.

## 4. 연속 질문이 어려운 이유

첫 질문과 후속 질문이 다음과 같다고 가정한다.

```text
첫 질문: 결제한 아이템이 지급되지 않았어요.
후속 질문: 그럼 얼마나 기다려야 해요?
```

후속 질문만 검색하면 `그럼`이 무엇을 의미하는지 알 수 없다. 검색 전에 다음과 같은 독립 질문으로
변환해야 한다.

```text
결제한 아이템이 지급되지 않은 경우 얼마나 기다려야 하나요?
```

이 과정을 query rewriting이라고 한다.

## 5. Bedrock Query Decomposition과의 차이

Knowledge Bases의 query decomposition은 복잡한 한 질문을 여러 하위 검색 질의로 분해하는
기능이다.

```text
아이템이 미지급됐고 중복 결제도 됐는데 어떻게 해야 하나요?
→ 아이템 미지급 대응 검색
→ 중복 결제 대응 검색
```

이 기능은 이전 대화를 기억하여 모호한 후속 질문을 독립 질문으로 바꾸는 conversational query
rewriting과 목적이 다르다.

## 6. 권장 연속 질문 구조

```text
chatbot-web
→ 최근 대화와 현재 질문 전송
→ chatbot-api Query Rewriter
→ 독립적인 검색 질문 생성
→ Retrieve
→ 점수 검사
→ 답변 생성 또는 고객센터 안내
```

요청 예:

```json
{
  "category": "payment",
  "question": "그럼 얼마나 기다려야 해요?",
  "history": [
    {
      "role": "user",
      "content": "결제한 아이템이 지급되지 않았어요."
    },
    {
      "role": "assistant",
      "content": "결제 내역과 상품 수령함을 확인해 주세요."
    }
  ]
}
```

재작성 결과:

```json
{
  "standaloneQuery": "결제한 아이템이 지급되지 않은 경우 얼마나 기다려야 하나요?"
}
```

## 7. 상태 관리 선택지

### Web이 최근 대화를 전송

장점:

- chatbot-api가 stateless하게 유지된다.
- Redis나 데이터베이스가 필요 없다.
- 토이프로젝트에 구현이 단순하다.

단점:

- 클라이언트가 보낸 이력을 신뢰하지 않고 길이와 형식을 검증해야 한다.
- 요청 크기를 제한해야 한다.

### 서버가 conversationId로 대화 저장

장점:

- 서버가 대화 이력을 일관되게 관리한다.
- 여러 기기와 장기 대화에 적합하다.

단점:

- Redis 또는 데이터베이스가 필요하다.
- 만료, 개인정보, 삭제 정책이 필요하다.

현재 구현은 문맥 오염과 비용을 줄이기 위해 직전 사용자 질문과 답변 한 쌍, 최대 2개 메시지만
Web이 전송하는 stateless 방식을 사용한다.

## 8. Query Rewriter 안전 규칙

재작성 모델에는 다음 규칙이 필요하다.

- 새로운 사실이나 답변을 추가하지 않는다.
- 현재 질문과 대화 이력만 사용한다.
- 질문의 카테고리를 변경하지 않는다.
- 검색 가능한 독립 질문 한 문장만 반환한다.
- 인사말이나 완전히 새로운 질문은 원문을 유지한다.

재작성 결과도 길이 제한과 문자열 검증을 수행해야 한다.

## 9. 비용 판단

후속 질문 재작성은 모델 호출이 하나 추가된다.

```text
일반 질문: Retrieve 1회 + Generate 1회
후속 질문: Rewrite 1회 + Retrieve 1회 + Generate 1회
```

비용을 줄이려면 첫 질문에는 재작성을 수행하지 않고, 실제 대화 이력이 존재할 때만 Nova Micro로
재작성한다.

## 10. 구현 순서

현재 구현된 범위:

1. Web UI를 메시지 목록 방식으로 변경했다.
2. ChatRequest에 최대 2개의 `history`를 추가했다.
3. Nova Micro가 `follow_up`과 `new_topic`을 판정한다.
4. `follow_up`이면 독립 질문으로 재작성하고 `new_topic`이면 현재 질문 원문을 사용한다.
5. 재작성 실패 시 현재 질문 원문을 사용한다.
6. 카테고리를 변경하면 Web 대화 이력을 초기화한다.

추가로 필요한 운영 작업:

1. 점수 기반 fallback을 실제 질문 데이터로 조정한다.
2. 원본 질문, 재작성 질문, 검색 점수와 선택 문서를 구조화해 로깅한다.
3. 모호한 후속 질문과 주제 전환 질문 테스트 세트를 작성한다.
